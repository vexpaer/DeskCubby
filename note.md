
请开发一款名为 DeskCubby 的安卓原生个人记录软件，使用 Kotlin 开发。整体界面采用底部五栏导航，默认包括：首页、日记、博客、闪思、设置。应用以本地优先为原则，核心记录功能在离线状态下也必须正常使用。

# 日记

日记功能需要与 Obsidian 的 Markdown 文件兼容。

设置中可以选择一个日记文件夹，DeskCubby 读取和修改其中的所有 `.md` 文件。Markdown 文件是日记内容的真实数据来源，不要把正文只保存在应用数据库中。

可以单独设置一个媒体文件夹。插入图片后，将图片复制到媒体文件夹中，并在当前 Markdown 光标位置自动插入相对路径引用。

日记编辑器支持：

- 编辑和保存 Markdown 文档
- 插入图片
- 图片预览
- 拖动图片块调整图片在文章中的位置
- 点击图片下方的名称修改图片说明
- 在设置中限制图片的最大显示宽度和高度
- 自动保存、撤销、重做
- 源码编辑模式和阅读预览模式
- 检测文件是否被 Obsidian 或其他软件从外部修改，避免直接覆盖

## 分类上传媒体

日记编辑器工具栏中需要有一个“上传媒体”按钮。

普通点击该按钮时，执行普通图片上传。

长按该按钮时，展开一个带动画的分类菜单，菜单中包含：

- 早餐
- 午餐
- 晚餐
- 水果
- 夜宵

点击其中一个分类后，打开系统图片选择器。选中的图片复制到媒体文件夹，并按照“日期 + 分类 + 序号”自动命名，例如：

```text
2026-07-18_早餐_01.jpg
2026-07-18_早餐_02.jpg
2026-07-18_晚餐_01.jpg
````

同一天、同一分类存在多张图片时，序号自动递增，不能覆盖已有图片。Markdown 中的图片说明默认使用对应分类，例如：

```markdown
![早餐](../Attachments/2026-07-18_早餐_01.jpg)
```

插入后仍然允许用户点击图片说明进行修改。分类菜单可以使用纵向浮动菜单、扇形菜单或围绕按钮展开的菜单，但必须便于单手操作，并且点击编辑器其他区域后自动收起。

## 日记组织方式

本地日记文件保持平铺存储，不创建月份子文件夹，例如：

```text
2026-01-06 去武汉玩.md
2026-01-08 实验记录.md
2026-02-03 开学.md
```

日记页面在界面上根据日期创建“虚拟月份文件夹”，例如：

```text
2026.07
12 篇日记

2026.06
18 篇日记
```

点击月份后显示该月的全部日记。长按日记可以重命名、修改标题、删除或恢复。

日记页面底部有“进入今日日记”按钮。点击后：

1. 查找今天对应的日记。
2. 已存在则直接打开。
3. 不存在则根据默认模板和文件名格式创建，再进入编辑器。

设置中可以配置：

* 日记文件名格式
* 日记标题格式
* 默认 Markdown 模板
* 日期格式
* 媒体文件夹
* 图片命名格式
* 图片最大显示尺寸

# 博客

博客页面是一个应用内浏览器，使用 Android WebView 实现。

需要支持常见浏览器功能：

* 地址栏
* 前进和后退
* 刷新
* 返回主页
* 网页内查找
* 收藏夹
* 浏览历史
* 文件上传和下载
* 使用外部浏览器打开
* 恢复上次浏览页面

设置中可以配置默认打开的网址。

# 闪思

闪思相当于快速备忘录，页面分为上下两个区域，中间有一条可以拖动的分界线。

上半部分显示闪思历史：

* 新记录靠近下方，旧记录向上排列
* 可以向上滑动查看更早的记录
* 每条记录在列表中最多显示一行
* 点击记录后在下方编辑
* 支持删除、置顶和复制

下半部分是编辑器：

* 未选中旧记录时，输入内容并点击发送会创建新记录
* 选中旧记录时，点击发送会更新该记录
* 支持取消当前选择

删除采用软删除。页面右上角提供回收站入口，可以查看、恢复或永久删除记录。

分界线位置需要自动保存，下次打开时恢复。

# 首页

首页采用可配置的小组件布局，用户可以添加、删除、拖动和调整小组件顺序。

首批支持：

* 日历
* 天气
* 每日诗词
* 今天日期
* 连续记录天数
* 本月日记数量
* 日记总字数
* 最近日记
* 最近闪思
* 快速输入
* 随机旧日记
* 年度进度
* 网站快捷入口

天气和每日诗词需要缓存上一次结果，断网时仍然可以显示。

# 设置

设置页面可以修改：

* 底部五个按钮的图标
* 按钮文字
* 按钮顺序
* 是否显示某个按钮
* 默认启动页面
* 应用主题
* 日记目录
* 媒体目录
* Markdown 模板
* 图片命名格式
* 浏览器默认网址
* 闪思回收站设置

设置页面始终需要保留至少一个可以进入设置的入口，避免用户将设置按钮隐藏后无法恢复。

# UI 风格

应用需要提供两套可以在设置中切换的完整视觉风格。切换风格后只改变视觉表现，不改变页面结构、数据和操作逻辑。

## Android 原生风格

使用 Jetpack Compose 和 Material 3 设计。整体强调清晰、简洁、实用，使用标准的顶部栏、底部导航栏、卡片、浮动按钮、对话框和 Material 图标。页面层级明确，间距统一，支持浅色、深色以及跟随系统主题。动画应简短自然，符合安卓原生应用的操作习惯。

## iOS 26 毛玻璃风格

提供一套接近 iOS 26 Liquid Glass 的视觉主题。底部导航栏、顶部工具栏、弹出菜单、小组件和编辑器浮动工具栏可以使用半透明毛玻璃、背景模糊、柔和高光、轻微边缘折射、圆角和分层阴影。

毛玻璃效果不能覆盖整个页面或影响文字可读性，应重点用于导航栏、工具栏、弹窗和悬浮组件。滚动或切换页面时可以有轻微的透明度、缩放和景深动画，但不要加入过度弹跳或高消耗特效。低性能设备需要自动降低模糊强度，必要时使用半透明纯色作为降级方案。

两套风格都应具有统一的尺寸、交互区域和信息架构，避免切换主题后出现组件错位。

# 技术路径

使用以下安卓原生技术栈：

* Kotlin
* Jetpack Compose
* Material 3
* Navigation Compose
* ViewModel
* StateFlow
* Room
* Preferences DataStore
* Storage Access Framework
* ContentResolver
* DocumentFile
* WorkManager
* Coil
* commonmark-java
* AndroidX WebKit
* Hilt

日记 Markdown 文件作为唯一真实数据来源，Room 只保存文件索引、统计数据、闪思、浏览历史、小组件布局和回收站数据。

文件夹授权使用 Storage Access Framework，并保存长期 URI 权限。不要尝试把 `content://` URI 强行转换为绝对文件路径，所有文件读写都通过 `ContentResolver` 和 `DocumentFile` 完成。

Markdown 编辑器第一阶段采用“源码编辑 + 阅读预览”方案，不要一开始开发完整所见即所得编辑器。图片拖动可以将独占一段的 Markdown 图片解析成图片块，拖动图片块时只移动对应 Markdown 片段。

两套 UI 风格都使用 Jetpack Compose 原生实现，不要使用网页或 WebView 模拟应用主界面。毛玻璃效果通过 Compose Blur、透明层、渐变高光和降级渲染实现，并关注低端设备性能。

先完成可运行的最小版本，再逐步增加复杂功能。推荐开发顺序为：

1. 应用骨架、底部导航和主题切换
2. SAF 文件夹授权和 Markdown 扫描
3. 日记列表、月份分组和今日日记
4. Markdown 编辑、保存和模板
5. 普通图片上传与分类图片上传
6. 闪思和回收站
7. WebView 浏览器
8. 首页小组件
9. 图片拖动、版本历史和外部修改冲突处理

代码需要保持清晰的分层结构，每完成一个阶段都进行编译和基础功能测试，不要一次性生成大量未经验证的代码。
